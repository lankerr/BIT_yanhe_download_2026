# m3u8 分片下载速度模型

> 目标：搞清楚"哪个旋钮拧一格能让下载快多少"，避免凭感觉优化。

---

## 一、变量定义

| 符号 | 含义 | 单位 |
|------|------|------|
| `N` | 分片总数 | 个 |
| `S` | 平均分片大小 | byte |
| `B_link` | 客户端到 CDN 的可用带宽 | byte/s |
| `B_per` | 单连接的稳态带宽（受 TCP 拥塞窗口限制） | byte/s |
| `RTT` | 客户端 ↔ CDN 往返时延 | s |
| `T_handshake` | 单连接 TCP+TLS 建连耗时 | s |
| `T_sign` | 客户端给 URL 算签名的耗时 | s |
| `K` | 并发线程数（连接数） | — |
| `p` | 单分片失败概率 | — |
| `T_retry_wait` | 失败重试间隔（指数退避基线） | s |
| `T_merge` | 合并 N 个 ts → 1 个 mp4 的耗时 | s |
| `T_total` | 端到端总耗时 | s |

延河的真实量级（我们看到的）：

- `N` ≈ 100 ~ 300（90 分钟课）
- `S` ≈ 200 KB ~ 2 MB（VGA 屏幕流单分片）
- `RTT`：校园网 ~10ms，VPN/校外 ~50ms+
- `B_link`：千兆校园网 ~125 MB/s 上限，实际 30-80 MB/s；VPN 通常 5-20 MB/s
- `B_per`：单 TCP 连接长肥管道下 ≈ `cwnd / RTT`，对延河 CDN 实测约 1-3 MB/s
- `T_handshake`：HTTP/1.1 + TLS ≈ 2 RTT ≈ 20-100ms

---

## 二、串行基线（K=1，无失败）

```
T_serial = T_sign·N + (T_handshake + S/B_per)·N
```

数量级：

```
N=297, S=500KB, B_per=2MB/s, T_handshake=50ms, T_sign≈1ms
T_serial ≈ 297 × (0.05 + 0.5/2 + 0.001) ≈ 297 × 0.301 ≈ 89.4 s
```

注意 `S/B_per = 0.25s` 占主导（83%），`T_handshake` 只占 17%。**所以 HTTP keepalive（消除 handshake）能省 17%，但消除不了 83% 的传输时间**——传输时间只能靠并发或带宽提升。

---

## 三、并发模型（K 线程，无失败）

理想化：K 条连接同时跑，每条独立享用 `B_per`，但**整体不能超过 `B_link`**：

```
B_eff(K) = min(K · B_per, B_link)

T_concurrent = (T_handshake + T_sign·N/K) + N·S / B_eff(K)
                                              ↑
                                       并发拉满后的传输时间
```

定义"饱和点" `K* = B_link / B_per`（再加线程也无收益）：

```
延河校园网： K* = 50MB/s ÷ 2MB/s = 25
延河 VPN：   K* = 10MB/s ÷ 2MB/s = 5
```

**关键洞察**：
- 校园网下 K=24 已经接近饱和，K=64 是浪费（多余的 39 个连接抢同一根管子）
- VPN 下 K=8 就饱和了，K=64 反而因为连接竞争 + 服务端限速 → 拖慢
- 这正好对得上业界共识：**N_m3u8DL-RE 默认 = CPU 线程数（通常 8-16），yt-dlp 默认是 1（!），aria2 默认 16**

### 加速比上界

```
Speedup(K) = T_serial / T_concurrent ≈ min(K, K*)
```

**实战推论**：从 K=4 升 K=24，加速 6 倍；从 K=24 升 K=64，加速 1 倍（无收益）。

## 四、失败与重试的代价

加入失败概率 `p`，每个分片期望尝试次数 `1/(1-p)`：

```
T_with_retry = T_concurrent × 1/(1-p) + N·p·T_retry_wait
                                          ↑
                                    指数退避总等待
```

延河实测 `p` 约 1-3%（VPN 下 5-10%）。代入：

```
N=297, p=2%, T_retry_wait=2s
惩罚 = 297 × 0.02 × 2 = 11.9s
传输部分 ×1.02 = +2%
总惩罚 ≈ 12s（在 90s 基线上 +13%）
```

**关键洞察**：失败惩罚的大头是**等待**而非重传。所以：
- ❌ 把 retry_wait 从 2s 加到 10s 一倍代价
- ✅ 把超时从 15s 改到 60s 让 `p` 从 2% 降到 0.5%（少 75% 失败）→ 总耗时 -10%

这就是为什么我们当前版本卡——**短超时人为制造了高 p**。

## 五、AIMD 的代价

类 TCP 的 +1/-2 窗口调整：

```
稳态窗口 W ≈ √(2/(p·a))    （Mathis 公式简化版，a=减系数）
```

代入 `p=2%, a=2`：W ≈ 7。**也就是说 AIMD 在 2% 失败率下平均只能维持 7 个连接，离 K*=24 的饱和点差 3 倍**。

这正是死亡螺旋的根因：你设的 `_max_workers=64` 是名义上限，AIMD 会把实际窗口压到 W≈7，吞吐降为饱和值的 1/3。

**结论**：固定 K=24 的吞吐 ≈ AIMD(0,64) 吞吐的 3 倍。

## 六、合并耗时

ts 文件用 MPEG-TS 容器，HLS 设计上**支持二进制 concat**（直接拼字节）：

```
T_binary_merge ≈ N·S / B_disk_write    // 顺序写盘，~500MB/s
T_ffmpeg_remux ≈ N·S / B_disk_write × 1.5  // 多一次 demux/remux 解析
T_ffmpeg_m3u8  ≈ N·S / B_disk_write × 2-3  // 还要解析 m3u8、按时间戳排序
```

297×500KB ≈ 150MB：
- 二进制 concat ≈ 0.3s
- ffmpeg copy mp4 ≈ 0.5s
- ffmpeg m3u8 协议（当前用法）≈ 1-2s

合并占比小（< 5%），但**正确性**比速度重要——后两者依赖外部 ffmpeg 行为，二进制 concat 几乎不可能出错。

## 七、把所有变量串起来

```
T_total ≈ max(N·S/B_link, N·S/(K·B_per))           ← 传输（取决于哪个先饱和）
        + (T_handshake + T_sign)·N/K                ← 单分片建连/签名（被并发摊薄）
        + N·p·T_retry_wait                          ← 失败惩罚
        + T_merge                                    ← 合并
```

把延河校园网的真实数代入（`B_link=50MB/s, B_per=2MB/s, K=24, p=2%`）：

```
N=297, S=500KB, T_handshake=50ms, T_sign=1ms, T_retry_wait=2s, T_merge=1s

传输：max(297×0.5/50, 297×0.5/(24×2)) = max(2.97, 3.10) = 3.10 s
建连：(0.05+0.001)×297/24 = 0.63 s
失败：297×0.02×2 = 11.88 s
合并：1 s
─────────────────────────────────
T_total ≈ 16.6 s     （理论下限）
```

而你当前实测大概 60-120s（VPN 下，加上 AIMD 死亡螺旋）。差出来的 4-7 倍**全是工程细节损耗**。

## 八、优化的优先级（按数学增益排序）

| 改动 | 数学预期 | 复杂度 |
|------|---------|--------|
| **超时 15s→60s 让 p 从 5% 降到 1%** | -8% ~ -20% | 1 行 |
| **删 AIMD，固定 K=24** | -50% ~ -67%（消除死亡螺旋） | 30 行 |
| **HTTP Session keepalive 复用连接** | -5% ~ -15%（省 handshake） | 5 行 |
| **二进制 concat 替代 ffmpeg m3u8** | -2% ~ -5% | 20 行 |
| **K 从 16 升到 24（饱和点）** | -10% ~ -30% | 1 行 |
| K 从 24 升到 64 | 0%（已饱和） | — |
| 改用 httpx + asyncio | -0% ~ +5%（Python 下 GIL 是瓶颈，async 不是银弹） | 200 行 |
| 改用 aria2 子进程 | -10% ~ -30%（省 GIL，C++ 实现） | 80 行 |
| 改用 N_m3u8DL-RE 子进程 | -10% ~ -30% | 60 行 |

**直觉验证**：前 3 项加起来理论上能从 90s 压到 30s 以内。

## 九、要做的对照实验

为了验证这套数学，我打算实测以下 5 个候选：

| ID | 实现 | 关键参数 |
|----|------|---------|
| `cur` | 当前 `m3u8dl.py` | 默认（AIMD 4-64） |
| `cur_fix` | 当前代码 + 阶段 1 三处修改 | 固定 K=24, timeout=(10,60), Lock 计数 |
| `req_pool` | 纯 `requests.Session + ThreadPoolExecutor` | K=24, keepalive |
| `aria2` | 子进程调用 aria2c | `-x16 -s16 -j16` 同时下载所有 ts |
| `n_m3u8dl_re` | 子进程调用 N_m3u8DL-RE | `--thread-count 24` |

每个候选用同一个 m3u8 URL（可以拿 67092 的那 297 个分片）跑 3 次，记录：

- `wall_clock_s`：端到端耗时
- `bytes_total`：实际下载字节数
- `throughput_mbps`：吞吐率
- `failed_count`：失败分片数
- `peak_memory_mb`：进程峰值内存
- `cpu_pct`：CPU 占用率峰值

报告会有：

- 折线图：每个候选 3 次 trial 的耗时
- 吞吐对比柱状图（按 50% / 95% percentile）
- 一张表：每个候选 vs 理论下限的 gap
